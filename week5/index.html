Ch 10: Testing and Debugging

Errors, Exceptions, and Warnings

  • Error - caused when something goes wrong
    • System error - something goes wrong with system or devices
    • Programmer error - syntax errors, faulty logic, typos
    • User error - user has entered data incorrectly (can also be considered programmer error programmers should anticipate and minimize user errors)
  • Exceptions - An error that returns a value that can be used to deal with the error
    • ReferenceError: unicorn is not defined
  • Stack Traces
    • Produced by exception
    • Works backward to find the start of the sequence that caused the exception
  • Warnings
    • Produced by errors that aren’t enough to cause the program to crash
    • Code still runs but possible incorrectly

Strict Mode

  • Produces more exceptions and warnings for debugging
  • Does not allow use of some deprecated features
    • Considers old practices in “poor style” and treats them as errors
  • Helps spot errors early on
  • For strict mode: ‘use strict’;
    • Can be used inside functions
    • Modules are in strict mode by default

Feature Detection

  • Use an if statement to see if a feature exists before trying to use it
    • Ensures no errors from trying to use a non-existent feature

Debugging in the browser

  • alert() - acts as breakpoint and stops code
  • console.log() - does not stop code, logs variable values
  • console.trace() - logs an interactive stack trace in the console
  • Browser debugging tools - set breakpoints, use ‘debugger’ keyword

Error Objects

  • Created when an exception occurs or with a constructor function for custom error objects
  • Throwing Exceptions
    • Throw your own exceptions using throw
      • throw new Error('Something has gone badly wrong!');

Exception Handling

  • Wrap code in a try block if you think it will result in an exception
  • A catch block catches the error
  • A finally block holds code that is executed after the try and catch even if no errors occur.

Tests

  • Test Driven Development (TDD)
    • The process 
      1. Write tests (that initially fail) 
      2. Write code to pass the tests 
      3. Refactor the code 
      4. Test refactored code 
      5. Write more tests for new features
