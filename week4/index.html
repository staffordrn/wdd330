Chapter 12

OOP -Juicer analogy

Encapsulation

  • Juicer makes juice - inner workings are hidden
    • Only essential functions are seen by user (on switch)

Polymorphism

  • Same process used for different objects (objects can share methods)
    • Juicer has on button and uses an electrical outlet but so so other things
    • It can juice lots of different fruits

Inheritance

  • Newer juicer model has all the functions of polder model and then some

Classes

  • Juicers made on a production line are all based on a prototype
    • A class would represent the design and the objects created based on that design are instances of the class

Chapter 15

Libraries

  • Reduces the amount of code you need to write
  • Makes common tasks easier
  • Thoroughly tested
  • Minimize the need for writing custom code
  • Aid productivity and consistency

When to use a library

  • Use a library with a modular structure so you only use what you need
  • Avoid libraries if you need only a little JavScript
  • Make common tasks easier
  • Try to find a library that has what you need and not more

*Try not to rely on the library so much that you’re not learning the language itself

Modular Javascript

  • Modular - self contained piece of code that can be reused
    • Single purpose
    • Good to use instead of large libraries when only a few modules are needed
  • Coupling - how much do two pieces of code depend on each other
    • Tightly coupled - very dependent (one relies on the other)
    • Loosely coupled - a piece of code can be easily swapped out for another
    • Good design - keep code loosely coupled as possible - flexibility
  • ES6 Modules
    • Allow you to keep parts of code in separate files
    • Module code is always strict
    • Modules have their own global scope (variables created in module are only accessible in module)
    • ‘this’ in top level is undefined rather than global object
    • No HTML-style comments in modules

MVC Frameworks

  • Models - objects that implement functionality for CRUD tasks
  • Views - provide a visual representation of the model and provide a way for r users to interact
  • Controllers - allow models and views to communicate
    • Respond to events
    • Process information
    • Update the model and view

*JavaScript does not always follow the controller pattern - MV* more accurately describes the various patterns JavaScript uses
